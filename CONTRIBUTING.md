# コントリビュートガイド

翻訳パックの投稿・改善を歓迎します。

## 新しいパックを投稿する

対象ライブラリを依存に持つプロジェクト(なければ空のプロジェクトに `npm install <パッケージ名>` するだけでも可)で、次を実行します。

```bash
# 1. 型定義から JSDoc を抽出する(既存パックがあれば訳を引き継ぐ)
npx yakudoc add <パッケージ名>

# 2. 翻訳する(どちらか)
npx yakudoc translate --engine local --pkg <パッケージ名>   # 内蔵モデル(オフライン)
npx yakudoc translate --engine prep  --pkg <パッケージ名>   # 任意の AI に依頼(prompt.md を貼る)

# 3. 共有用に書き出す
npx yakudoc export <パッケージ名>
```

出力された `<パッケージ名>.json` を、このリポジトリの `packs/<言語コード>/<パッケージ名>.json` に置いてプルリクエストしてください。

- スコープ付きパッケージのファイル名は `/` を `__` に置換します(例: `@types/node` → `@types__node.json`)
- 言語コードは翻訳先に合わせます(`ja` `ko` `zh` `de` `fr` …)

## 既存のパックを改善する

訳文がおかしい・古い・抜けている場合は、該当の `packs/<言語>/<パッケージ名>.json` の `translated` を直接編集して PR を送ってください。`entries` のキー(ハッシュ)や `original` は変更しないでください(照合が壊れます)。

## 翻訳の指針

- JSDoc の一行説明として自然な文体に(日本語なら「です・ます調」で簡潔に)
- コード片・`{@link ...}`・型注釈・URL は `<ph0>` のような保護トークンになっています。**翻訳・削除せず、対応する位置にそのまま残してください**(yakudoc のツールが自動で保護・復元します)
- 固有名詞や広く使われる技術用語(callback、Promise など)は無理に訳さずカタカナ・原語でよいです

## 検証

PR を送ると、CI が各パックの JSON 形式を検証します(`name` / `lang` / `entries` の有無、`entries` の各要素が `original` を持つか、ファイル名と `name` の整合、ディレクトリ名と `lang` の整合)。ローカルでも実行できます。

```bash
node scripts/validate.mjs
```
